**Fix 6 — InventoryRoomForm.tsx line 205: Select onValueChange type**

The `handleConditionChange` function has the signature `(val: string) => void` which matches Shadcn Select's `onValueChange` prop type. If TypeScript reports a type error here, it is likely because the inferred type of `Select`'s `onValueChange` expects the handler to explicitly be typed as `(value: string) => void`. Apply an explicit type annotation to the handler:

```typescript
const handleConditionChange = (val: string): void => {
  const c = val as RoomCondition;
  setCondition(c);
  notify({ condition: c });
};
```

If the error is on the `onValueChange={handleConditionChange}` JSX attribute instead, cast inline:
```tsx
onValueChange={handleConditionChange as (value: string) => void}
```

**Fix 9 — properties/add/page.tsx: Resolver type mismatch with z.coerce.number()**

`z.coerce.number()` produces a schema whose inferred TypeScript type is `number`, but `zodResolver` with react-hook-form sometimes infers the form field values as `string` (because HTML inputs always return strings). The mismatch causes the `Resolver<AddPropertyFormData>` type to fail.

The cleanest fix: keep `z.coerce.number()` in the schema (so Zod handles coercion from the string input) and cast the resolver:

```typescript
resolver: zodResolver(addPropertySchema) as Resolver<AddPropertyFormData>,
```

Import `Resolver` from `"react-hook-form"`:
```typescript
import { useForm, type Resolver } from "react-hook-form";
```

**Fix 10 — tools/yield-calculator/page.tsx: same Resolver type mismatch**

Apply the identical fix as Fix 9. The schema uses `z.coerce.number()` for all numeric fields. Cast the resolver:

import { useForm, type Resolver } from "react-hook-form";

**Fix 11 — services/landlord/document-service.ts line 248: unsafe type cast**

```typescript
const property = (doc.property as { address_line_1: string; city: string; postcode: string } | null) ?? { address_line_1: "", city: "", postcode: "" };
```

This is an unsafe cast that TypeScript may flag depending on the Supabase return type. Replace the inline cast with a safe extraction using optional chaining and unknown intermediate type:

```typescript
const rawProperty = doc.property as unknown;

  typeof rawProperty === "object" &&
  "address_line_1" in rawProperty
    ? (rawProperty as { address_line_1: string; city: string; postcode: string })
    : { address_line_1: "", city: "", postcode: "" };
```

This satisfies the TypeScript compiler without changing the runtime behaviour.
