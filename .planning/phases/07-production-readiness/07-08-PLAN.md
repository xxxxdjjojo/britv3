  <files>britv3.0/tests/security/rls-audit.sql, britv3.0/tests/security/rls-audit.test.ts, britv3.0/docs/rls-audit-report.md</files>
  <action>
1. Create `tests/security/rls-audit.sql`:
   - Query 1: Find all tables in public schema WITHOUT RLS enabled:
     ```sql
     SELECT schemaname, tablename
     FROM pg_tables
     WHERE schemaname = 'public'
     AND tablename NOT IN (
       SELECT relname FROM pg_class WHERE relrowsecurity = true
     );
     ```
   - Query 2: List all RLS policies per table:
     ```sql
     SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual
     FROM pg_policies
     WHERE schemaname = 'public'
     ORDER BY tablename, policyname;
     ```
   - Query 3: Check for tables with RLS enabled but no policies (security hole):
     ```sql
     SELECT c.relname
     FROM pg_class c
     JOIN pg_namespace n ON c.relnamespace = n.oid
     WHERE n.nspname = 'public'
     AND c.relkind = 'r'
     AND c.relrowsecurity = true
     AND c.relname NOT IN (
       SELECT tablename FROM pg_policies WHERE schemaname = 'public'
     );
     ```

2. Create `tests/security/rls-audit.test.ts`:
   - This is an integration test that runs against the Supabase database
   - Test: "All public tables have RLS enabled" -- run query 1, assert result is empty
   - Test: "All RLS-enabled tables have at least one policy" -- run query 3, assert result is empty
   - Test: "Known sensitive tables have RLS" -- explicitly check profiles, properties, messages, bookings, push_subscriptions, content_reports have RLS
   - Use supabase admin client (service_role) to execute raw SQL via `.rpc('exec_sql', { sql })` or direct pg connection if available
   - Skip tests gracefully if database connection not available (CI-safe)
