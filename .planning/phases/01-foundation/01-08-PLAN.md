    - exportUserData(userId) queries profiles, user_roles, consent_records, consent_audit_log, auth_audit_log and returns JSON object
    - POST /api/gdpr/delete creates deletion_request with 30-day purge date, returns confirmation

   - `consent.test.ts`: initializeConsent creates 3 records; updateConsent upserts correctly; getConsent returns all types
   - `export.test.ts`: exportUserData returns correct JSON structure with all user tables; handles user with no data gracefully
   - `deletion.test.ts`: deletion request creates with correct purge date (30 days); prevents duplicate pending requests
   - `audit.test.ts`: Verify consent_audit_log structure matches expected format (this tests the TypeScript types/service assumptions -- the actual trigger runs in Postgres)

2. Create `src/services/gdpr/consent-service.ts`:
   - `initializeConsent(userId, consents: Record<ConsentType, boolean>, ip: string, userAgent: string)` -- inserts 3 consent_records (marketing, analytics, third_party). Uses Supabase server client. The DB trigger automatically logs to consent_audit_log.
   - `getConsent(userId: string)` -- select from consent_records WHERE user_id
   - `updateConsent(userId, consentType: ConsentType, granted: boolean, ip: string, userAgent: string)` -- upsert consent_record. The DB trigger handles audit logging.
   - All operations capture ip_address and user_agent from request headers for audit trail compliance.

3. Create `src/services/gdpr/export-service.ts`:
   - `exportUserData(userId: string)` -- queries via admin client (bypasses RLS for complete export):
     * profiles (own row)
     * user_roles (own roles)
     * consent_records (own consents)
     * consent_audit_log (own audit trail)
     * auth_audit_log (own auth events)
     * provider_verifications (own verifications, if any)
     * deletion_requests (own requests, if any)
   - Returns structured JSON: `{ exported_at, user_id, profile, roles, consent, consent_history, auth_events, verifications, deletion_requests }`
   - Excludes internal IDs and system fields not relevant to user

4. Create `src/app/api/gdpr/export/route.ts`:
   - GET handler
   - Verify auth via `supabase.auth.getUser()`
   - Call `exportUserData(user.id)`
   - Return Response with `Content-Type: application/json`, `Content-Disposition: attachment; filename="britestate-data-export-{date}.json"`
   - Log export event to auth_audit_log

5. Create `src/app/api/gdpr/delete/route.ts`:
   - POST handler
   - Verify auth via `supabase.auth.getUser()`
   - Check for existing pending deletion request (prevent duplicates)
   - Insert into deletion_requests with `scheduled_purge_at = NOW() + 30 days`
   - Log deletion request to auth_audit_log
   - Return 200 with `{ scheduled_purge_at, message: "Account scheduled for deletion" }`
   - Note: Actual data purge will be handled by a scheduled function (Supabase Edge Function or pg_cron) -- out of scope for this plan, just the request mechanism
