    **`britv3.0/src/services/listings/listing-service.ts`:**
    Pure async functions following Phase 1 pattern:

    - `createListing(userId: string, input: CreatePropertyInput & CreateListingInput)`:
      1. Geocode postcode via geocode-service to get lat/lng
      2. Insert into `properties` table (address, type, beds, baths, features, etc.) with `ST_MakePoint(lng, lat)` for coordinates
      3. Insert into `listings` table (property_id, user_id, listing_type, price, status='draft', etc.)
      4. Return the created listing with property data
      5. Fire-and-forget: refresh materialized view

    - `updateListing(userId: string, listingId: string, input: UpdatePropertyInput & UpdateListingInput)`:
      1. Verify ownership: query listing where id=listingId AND user_id=userId
      2. Update properties table fields if property fields changed
      3. Update listings table fields if listing fields changed (price change triggers DB trigger for price_history)
      4. Fire-and-forget: refresh materialized view
      5. Return updated listing

    - `getListing(listingId: string)`: Join listings + properties + media, return full detail
    - `getListingBySlug(slug: string)`: Same join but by slug column
    - `getMyListings(userId: string, params: { status?, page? })`: Paginated list of user's listings
    - `deleteListing(userId: string, listingId: string)`: Soft delete (set deleted_at)
    - `getPriceHistory(listingId: string)`: Query price_history ordered by changed_at DESC
    - `getListingAnalytics(listingId: string)`: Return view_count, favorite_count, enquiry_count from listing
    - `incrementViewCount(listingId: string)`: RPC or direct update to increment view_count

    For PostGIS coordinate insertion, use RPC or raw SQL: `supabase.rpc('set_property_coordinates', { property_id, lng, lat })`. Create this RPC function in the migration if not already present, or use `supabase.from('properties').update({ coordinates: \`SRID=4326;POINT(${lng} ${lat})\` })`. Note: Supabase JS client may not directly support GEOGRAPHY types -- check and use `.rpc()` if needed.

  <behavior>
    Client compression:
    - compressImage(5MB file) returns file < 1MB
    - compressImage(400KB file) returns original (skip small files)
    - compressMultiple([files]) compresses all in parallel with progress callback

    Server processing:
    - processPropertyImage(buffer) validates MIME type via magic bytes
    - processPropertyImage rejects non-image files (PDF disguised as .jpg)
    - processPropertyImage strips EXIF metadata
    - processPropertyImage converts to WebP
    - processPropertyImage generates 400x300 thumbnail

    File validation:
    - validateImageFile accepts JPEG, PNG, WebP
    - validateImageFile rejects files > 10MB
    - validateDocumentFile accepts PDF, JPEG, PNG for floor plans
    - validateDocumentFile rejects other file types

    Media service:
    - uploadPropertyImage stores processed + thumbnail in Supabase Storage
    - uploadPropertyImage creates property_media record with sort_order
    - Enforces max 30 images per listing
    - deletePropertyImage removes from storage and database
    - reorderMedia updates sort_order for all media of a listing
  </behavior>
  <action>
    **`britv3.0/src/lib/upload/compress.ts`:**
    Follow research Pattern 2 exactly. Client-side module using browser-image-compression:
    - `compressImage(file: File): Promise<File>` -- skip if < 500KB, compress to 1MB max, 2400px max dimension, WebP, quality 0.85
    - `compressMultiple(files: File[], onProgress?): Promise<File[]>` -- parallel compression

    **`britv3.0/src/lib/upload/validate.ts`:**
    - `validateImageFile(buffer: Buffer): Promise<{ valid: boolean; mime: string; error?: string }>` -- uses `file-type` for magic byte detection, allows image/jpeg, image/png, image/webp, rejects > 10MB
    - `validateDocumentFile(buffer: Buffer): Promise<...>` -- allows above + application/pdf
    - `ALLOWED_IMAGE_TYPES` and `ALLOWED_DOCUMENT_TYPES` constants
    - `MAX_IMAGE_SIZE = 10 * 1024 * 1024` (10MB)

    **`britv3.0/src/lib/upload/process.ts`:**
    Follow research Pattern 4 exactly. Server-side module using sharp:
    - `processPropertyImage(buffer: Buffer): Promise<{ processed: Buffer; thumbnail: Buffer }>` -- auto-rotate, strip EXIF, resize to fit 2400x1800, WebP quality 85, thumbnail 400x300 cover crop WebP quality 75

    **`britv3.0/src/services/listings/media-service.ts`:**
    - `uploadPropertyImage(userId: string, listingId: string, file: Buffer, filename: string)`:
      1. Verify user owns listing
      2. Check count: query property_media WHERE listing_id -- reject if >= 30
      3. Validate file via `validateImageFile`
      4. Process via `processPropertyImage` (get processed + thumbnail buffers)
      5. Generate storage path: `property-images/${listingId}/${crypto.randomUUID()}.webp`
      6. Upload processed to Supabase Storage `property-images` bucket
      7. Upload thumbnail to same bucket with `-thumb` suffix
      8. Insert `property_media` record with url, thumbnail_url, media_type='image', sort_order = (max existing + 1)
      9. Return media record

    - `uploadFloorPlan(userId: string, listingId: string, file: Buffer, filename: string)`:
      1. Similar flow but validates as document, stores in `property-documents` bucket
      2. media_type = 'floor_plan'
      3. No thumbnail generation for PDFs (only for images)

    - `deletePropertyImage(userId: string, mediaId: string)`:
      1. Verify ownership via listing -> user_id
      2. Remove files from storage
      3. Delete property_media record

    - `reorderMedia(userId: string, listingId: string, mediaIds: string[])`:
      1. Verify ownership
      2. Update sort_order for each media ID based on array index

    **`britv3.0/src/app/api/listings/[id]/media/route.ts`:**
    - POST: Accept multipart form data, extract file buffer, call uploadPropertyImage or uploadFloorPlan based on `type` field
    - DELETE: Accept mediaId in body, call deletePropertyImage
    - PATCH: Accept ordered mediaIds array, call reorderMedia
    - All operations require auth + listing ownership
    - Set `export const maxDuration = 30;` for processing time on Vercel

    **Tests:** Write tests FIRST. Mock sharp (vi.mock("sharp")), mock file-type, mock Supabase Storage. Assert validation, processing calls, storage paths, database records.
