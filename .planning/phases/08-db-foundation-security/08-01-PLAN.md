CREATE POLICY "buyer_documents_insert" ON storage.objects

**Section 2 — 10 new tables (create in FK-dependency order):**

1. `viewing_slots` — created by agents, booked by buyers
2. `viewings` — buyer's booked viewing (references viewing_slots)
3. `offers` — buyer's offer on a listing
4. `offer_status_history` — immutable audit trail for offer state machine
5. `user_documents` — buyer's uploaded identity/funds/AIP documents
6. `ai_match_preferences` — buyer's AI match criteria (one row per user, UNIQUE on user_id)
7. `ai_match_results` — cached AI match scores, expire after 24 hours
8. `moving_checklist_items` — per-user per-offer checklist items
9. `referral_codes` — one permanent code per user (UNIQUE on user_id AND code)
10. `referral_conversions` — tracks who signed up via which code

Use the exact schema from 08-RESEARCH.md Code Examples section. Key constraints to implement exactly:
- `offers.status` CHECK: `('submitted', 'solicitors_instructed', 'searches', 'survey', 'mortgage_approved', 'exchange', 'completion', 'withdrawn')`
- `viewings.status` CHECK: `('confirmed', 'cancelled', 'rescheduled', 'completed')`
- `viewing_slots.status` CHECK: `('available', 'booked', 'cancelled')`
- `viewing_slots.type` and `viewings.type` CHECK: `('in_person', 'virtual')`
- `ai_match_results.match_score` CHECK: `BETWEEN 0 AND 1`
- `referral_codes`: UNIQUE on `user_id` AND separate UNIQUE on `code`
- `offer_status_history`: no `updated_at` — this is an immutable append-only log

Enable RLS on every table immediately after CREATE TABLE:
`ALTER TABLE public.{table_name} ENABLE ROW LEVEL SECURITY;`

**Section 3 — RLS policies:**

For `viewing_slots`:
- SELECT: `auth.uid() = agent_id` (agent sees their own slots)
- INSERT: `auth.uid() = agent_id`
- UPDATE: `auth.uid() = agent_id`

For `viewings`:
- SELECT: `auth.uid() = user_id OR auth.uid() IN (SELECT user_id FROM public.listings WHERE id = listing_id)`
  (buyer sees their bookings; listing owner sees viewings on their listing)
- NOTE: INSERT and UPDATE on viewings go through the `book_viewing_slot` SECURITY DEFINER RPC only — no direct client INSERT policy needed

For `offers`:
- SELECT: `auth.uid() = user_id OR auth.uid() IN (SELECT user_id FROM public.listings WHERE id = listing_id)`
- INSERT: `auth.uid() = user_id` (buyer can submit their own offer)
- UPDATE: `false` — blocked entirely; all status transitions via service role API routes

For `offer_status_history`:
- No INSERT policy (service role bypasses RLS; client INSERT is blocked by default-deny)
- SELECT (buyer): `auth.uid() IN (SELECT user_id FROM public.offers WHERE id = offer_id)`
- SELECT (listing owner): `auth.uid() IN (SELECT user_id FROM public.listings WHERE id = (SELECT listing_id FROM public.offers WHERE id = offer_id))`

For `user_documents`:
- SELECT: `auth.uid() = user_id`
- INSERT: `auth.uid() = user_id`
- UPDATE: `auth.uid() = user_id`
- DELETE: `auth.uid() = user_id`

For `ai_match_preferences`, `ai_match_results`, `moving_checklist_items`:
- Full CRUD: `auth.uid() = user_id` for all operations

For `referral_codes`:
- SELECT: `auth.uid() = user_id`
- INSERT: `auth.uid() = user_id`
- No UPDATE or DELETE (codes are permanent)

For `referral_conversions`:
- SELECT: `auth.uid() = referrer_id OR auth.uid() = referred_id`
- INSERT: service role only (no client INSERT policy — sign-ups go through server-side API)

**Section 5 — RPCs (SECURITY DEFINER functions):**

Create three RPCs as specified in 08-CONTEXT.md and 08-RESEARCH.md:

`book_viewing_slot(p_slot_id uuid, p_user_id uuid, p_listing_id uuid, p_type text)`:
- Opens a transaction
- `SELECT * FROM public.viewing_slots WHERE id = p_slot_id FOR UPDATE`
- If `v_slot.status <> 'available'` → RETURN `jsonb_build_object('success', false, 'error', 'SLOT_UNAVAILABLE')`
- Validate `p_user_id = auth.uid()` as first security check inside the function
- `UPDATE viewing_slots SET status = 'booked', updated_at = now() WHERE id = p_slot_id`
- `INSERT INTO viewings (user_id, slot_id, listing_id, type) VALUES (...) RETURNING * INTO v_viewing`
- RETURN `jsonb_build_object('success', true, 'viewing_id', v_viewing.id)`

`cancel_viewing(p_viewing_id uuid)`:
- SELECT viewing WHERE id = p_viewing_id AND user_id = auth.uid() FOR UPDATE
- If not found → RETURN `jsonb_build_object('success', false, 'error', 'NOT_FOUND')`
- UPDATE viewings SET status = 'cancelled', updated_at = now()
- UPDATE viewing_slots SET status = 'available', updated_at = now() WHERE id = v_viewing.slot_id
- RETURN `jsonb_build_object('success', true)`

`reschedule_viewing(p_viewing_id uuid, p_new_slot_id uuid)`:
- SELECT viewing WHERE id = p_viewing_id AND user_id = auth.uid() FOR UPDATE
- SELECT new slot WHERE id = p_new_slot_id FOR UPDATE
- If new slot not available → RETURN `jsonb_build_object('success', false, 'error', 'SLOT_UNAVAILABLE')`
- UPDATE old slot → status = 'available'
- UPDATE new slot → status = 'booked'
- UPDATE viewing → status = 'rescheduled', slot_id = p_new_slot_id
- RETURN `jsonb_build_object('success', true)`

All three RPCs: `LANGUAGE plpgsql SECURITY DEFINER`.
