**Provider service (src/services/marketplace/provider-service.ts):**

All functions accept a Supabase client as first parameter (server or client depending on context).

Export these functions:

1. `createProviderProfile(supabase, userId, data: ProviderProfileInput)` -- INSERT into service_provider_details. Generate slug from business_name (lowercase, hyphenated, unique check). Use geocodePostcode for the first service_postcode to set base_location. Returns the created record.

2. `updateProviderProfile(supabase, userId, data: Partial<ProviderProfileInput>)` -- UPDATE where user_id matches. Regenerate slug if business_name changed. Returns updated record.

3. `getProviderBySlug(supabase, slug: string)` -- SELECT with JOIN to profiles and provider_rating_stats. Returns full provider profile with rating stats or null.

4. `getProviderProfile(supabase, userId: string)` -- SELECT own profile by user_id.

5. `searchProviders(supabase, params: ProviderSearchInput)` -- Call the `search_providers()` RPC function with parameters. If postcode provided, geocode it first via postcodes-io service. Returns paginated results.

6. `uploadProviderDocument(supabase, userId, file: Buffer, documentType, originalFilename)` -- Use file-validator to validate magic bytes. Upload to Supabase Storage bucket "provider-docs" at path `provider-documents/{userId}/{documentType}/{timestamp}.{ext}`. INSERT into provider_documents table. Update profiles.provider_verification_status to 'pending_review' if currently 'unverified'. Returns document record.

7. `getProviderDocuments(supabase, userId: string)` -- SELECT all documents for provider. Admins can view any provider's docs (handled by RLS).

Use Zod schemas from marketplace-schemas.ts for input validation where appropriate.

**Search route (src/app/api/providers/search/route.ts):**
- GET handler
- Parse query params: service_category, postcode, radius, min_rating, search_query, limit (default 20), offset (default 0)
- Validate with providerSearchSchema
- Create Supabase server client
- Call searchProviders from provider-service
- Return JSON array with 200, or 400 for validation errors
- No auth required (public endpoint)

**Provider profile route (src/app/api/providers/[slug]/route.ts):**
- GET handler
- Extract slug from params
- Create Supabase server client
- Call getProviderBySlug
- Return 200 with provider data or 404
- No auth required (public endpoint)

**Document upload route (src/app/api/providers/documents/upload/route.ts):**
- POST handler, requires auth
- Parse multipart form data: file (required), document_type (required)
- Validate document_type against Zod schema
- Get current user from Supabase auth
- Convert file to Buffer
- Call uploadProviderDocument from provider-service
- Return 201 with document record or appropriate error (400 for validation, 401 for unauth, 413 for too large)
- Rate limit consideration: max 10 uploads per hour (document in comments, enforce later)

All routes follow Next.js App Router patterns: export async function GET/POST(request: NextRequest).
Return NextResponse.json() with appropriate status codes.
Handle errors with try/catch, return structured { error: string } on failure.
