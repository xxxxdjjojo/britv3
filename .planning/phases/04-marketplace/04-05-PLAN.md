**Booking service (src/services/marketplace/booking-service.ts):**

1. `createBooking(supabase, userId, data: BookingCreateInput)` -- Validate with bookingCreateSchema. Verify the quote exists, is accepted, and belongs to user's RFQ. Check for date conflicts with `checkDateConflict`. INSERT into bookings with status='pending_confirmation'. Booking reference auto-generated by DB trigger. INSERT initial status into booking_status_history. Returns created booking.

2. `getBooking(supabase, bookingId: string)` -- SELECT with JOINs to service_requests, quotes, profiles (user and provider). Include booking_status_history ordered by created_at. RLS handles access.

3. `listBookings(supabase, userId, role: "user" | "provider", status?: BookingStatus, limit=20, offset=0)` -- If role=user, filter by user_id. If role=provider, filter by provider_id. Ordered by scheduled_start_date DESC. Return paginated results with booking counts by status.

4. `updateBookingStatus(supabase, userId, bookingId, newStatus: BookingStatus, reason?: string)` -- Load current booking. Determine actor role (user if userId matches booking.user_id, provider if matches booking.provider_id, system for admin). Use `canTransition()` from booking-state-machine to validate. If not allowed, throw error with details. If requires_reason and no reason provided, throw error. UPDATE booking status. INSERT into booking_status_history with from_status, to_status, changed_by, reason. If newStatus='completed', the DB trigger increments provider's completed_jobs_count. Return updated booking.

5. `checkDateConflict(supabase, providerId, startDate: Date, endDate: Date, excludeBookingId?: string)` -- SELECT from bookings WHERE provider_id matches AND status IN ('confirmed', 'in_progress') AND daterange overlaps [startDate, endDate]. Also check provider_availability for unavailable periods. Exclude the given bookingId if provided (for rebooking). Returns { hasConflict: boolean; conflictingBookings: Array<{ id, start, end }> }.

6. `setProviderAvailability(supabase, providerId, startDate: Date, endDate: Date, reason?: string)` -- INSERT into provider_availability. Validate endDate > startDate.

7. `getProviderAvailability(supabase, providerId, fromDate?: Date, toDate?: Date)` -- SELECT from provider_availability, optionally filtered by date range. Also SELECT confirmed/in_progress bookings in range. Returns both unavailable periods and booked periods.

**Booking routes:**

1. `POST /api/bookings/create` -- Auth required. Parse JSON body. Call createBooking. Return 201 or 409 if date conflict.

2. `GET /api/bookings/list` -- Auth required. Parse query params: role ("user"/"provider"), status, limit, offset. Call listBookings. Return 200 with paginated results.

3. `GET /api/bookings/[id]` -- Auth required. Call getBooking. Return 200 with full booking detail including status history, or 404.

4. `PATCH /api/bookings/[id]/status` -- Auth required. Parse JSON body: { status: BookingStatus, reason?: string }. Call updateBookingStatus. Return 200 on success, 400 if invalid transition (include error message explaining why), 403 if wrong role for this transition.

**Availability route:**

5. `GET /api/providers/availability` -- Auth required (provider). Parse query params: from_date, to_date. Call getProviderAvailability. Return 200 with unavailable periods and booked periods.

5b. `POST /api/providers/availability` -- Auth required (provider). Parse JSON body: { start_date, end_date, reason }. Call setProviderAvailability. Return 201.

Combine GET and POST in the same route file.

All routes: structured error responses, appropriate HTTP status codes. The PATCH status endpoint should return a clear error message when transitions are invalid (e.g., "Cannot transition from 'pending_confirmation' to 'completed'. Valid next statuses: confirmed, cancelled").
