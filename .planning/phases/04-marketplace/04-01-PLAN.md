Create the complete marketplace migration following the epic4final.md spec exactly. The migration must include:

**Extensions** (idempotent):

- `service_category` (16 values: conveyancing, surveying, mortgage_broker, moving_company, home_inspector, cleaning, handyman, plumber, electrician, landscaping, interior_design, architect, property_management, pest_control, locksmith, other)

- `document_verification_status` (4 values: pending, approved, rejected, more_info_required)
- `rfq_status` (5 values: open, quotes_received, awarded, cancelled, expired)
- `quote_status` (7 values: draft, sent, viewed, accepted, declined, expired, withdrawn)
- `booking_status` (6 values: pending_confirmation, confirmed, in_progress, completed, cancelled, disputed)

**Tables (13) with columns exactly matching the epic spec Section 5.1:**
1. `service_provider_details` -- business info, services array (service_category[]), service_postcodes TEXT[], base_location GEOGRAPHY(Point,4326), pricing JSONB, slug UNIQUE, completed_jobs_count, years_in_business. UNIQUE on user_id.
2. `provider_documents` -- user_id, document_type, file_name, file_url, file_size, mime_type, verification_status, expiry_date, reviewer notes
3. `provider_availability` -- provider_id, start_date DATE, end_date DATE, reason TEXT
4. `service_requests` (RFQs) -- user_id, service_category, title, description, property_address, property_postcode, property_location GEOGRAPHY(Point,4326), preferred_start_date, urgency_level, budget_min/max, attachments JSONB, status, expires_at, view_count, quote_count
5. `quotes` -- service_request_id, provider_id, quote_number, total_amount, vat_included, line_items JSONB, scope_of_work, estimated_duration, payment_terms, warranty_info, validity_date, status, version
6. `bookings` -- service_request_id, quote_id, user_id, provider_id, booking_reference, scheduled_start_date, scheduled_end_date, actual_start_date, actual_end_date, status, cancellation_reason, cancelled_by
7. `booking_state_transitions` -- lookup table with from_status, to_status, allowed_by TEXT[], requires_reason BOOLEAN. Seed all 9 valid transitions.
8. `booking_status_history` -- booking_id, from_status, to_status, changed_by UUID, reason, created_at
9. `reviews` -- booking_id (UNIQUE), provider_id, reviewer_id, overall_rating, punctuality_rating, quality_rating, value_rating, professionalism_rating, title, review_text, search_vector tsvector, sentiment, authenticity_score, fake_review_probability, spam_indicators JSONB, moderation_status, provider_response, provider_response_at, helpful_count, not_helpful_count, deleted_at
10. `review_helpfulness` -- review_id, user_id, is_helpful BOOLEAN. UNIQUE on (review_id, user_id).
11. `review_flags` -- review_id, user_id, reason (spam, inappropriate, fake, off_topic, contact_info, promotional, duplicate), description, admin_status, reviewed_by, reviewed_at
12. `moderation_queue` -- review_id, priority_score, assigned_to, assigned_at, completed_at, decision, reason
13. `provider_rating_stats` -- provider_id PK, average_rating, total_reviews, dimensional averages, star distribution counts, helpful_votes, response_rate, last_review_date

**Triggers (6):**
1. `update_updated_at_column` reuse or create for all marketplace tables
2. `generate_quote_number` -- on quotes INSERT, format: QT-YYYYMMDD-XXXXX (random 5-digit)
3. `generate_booking_reference` -- on bookings INSERT, format: BK-YYYYMMDD-XXXXX
4. `update_completed_jobs_count` -- on bookings UPDATE when status -> 'completed'
5. `update_provider_rating_stats_incremental` -- on reviews UPDATE when moderation_status -> 'approved'
6. `on_review_created` -- on reviews INSERT, calculate authenticity_score and fake_review_probability

**Partial unique index:**
- `CREATE UNIQUE INDEX ON quotes(service_request_id) WHERE status = 'accepted';` (prevents race condition on double-accept)

**UK postcode validation constraint:**
- CHECK constraint on service_requests.property_postcode matching UK postcode regex

**Key indexes** (from epic spec Section 5.1):
- GIN on service_provider_details.services
- GiST on service_provider_details.base_location
- GIN on service_provider_details.service_postcodes
- btree on service_provider_details.slug
- btree on service_requests(user_id, status)
- btree on service_requests(service_category, status) WHERE status = 'open'
- GiST on service_requests.property_location
- btree on quotes(service_request_id, status)
- btree on quotes(provider_id, status)
- btree on bookings(user_id, status)
- btree on bookings(provider_id, status)
- btree on bookings(scheduled_start_date) WHERE status IN ('confirmed', 'in_progress')
- btree on reviews(provider_id, moderation_status) WHERE deleted_at IS NULL
- GIN on reviews.search_vector

**RLS policies:**
Follow the epic spec Section 5.1 RLS policy definitions. Enable RLS on ALL tables. Key policies:
- service_provider_details: Public SELECT for verified providers; provider manages own row
- provider_documents: Provider manages own; admins can view all
- service_requests: Owner manages own; verified providers can SELECT matching open RFQs
- quotes: Provider manages own; RFQ owner can SELECT quotes for their RFQs
- bookings: Both user and provider can SELECT their bookings; users INSERT
- reviews: Public SELECT for approved; reviewer INSERT for own completed bookings
- review_helpfulness/flags: Users manage own
- moderation_queue: Only is_admin=true users

**search_providers() function:**
Include the full `search_providers()` SQL function from epic spec Section 5.2 exactly as written.

Follow the same SQL style as 001_foundation.sql: section headers with comment blocks, consistent spacing.
