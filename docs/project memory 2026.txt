## 2. Architecture Decisions

### 2.1 ADR-001: Next.js App Router

**Decision:** Use Next.js 15 with App Router (not Pages Router)

**Context:**
- Need for SEO optimization (SSR/SSG)
- Complex routing requirements (6+ user roles)
- Server Components for performance
- API routes for backend functionality

**Rationale:**
- App Router provides better data fetching patterns
- Server Components reduce client-side JavaScript
- Nested layouts for dashboard structures
- Built-in API routes eliminate separate backend

**Trade-offs:**
- Learning curve for team (App Router is newer)
- Async params migration required (Next.js 15)
- Some third-party libraries need adaptation

**Consequences:**
- All routes use `/app` directory
- Server Components are default
- Client Components marked with `"use client"`
- API routes in `/app/api/**/route.ts`

### 2.2 ADR-002: Supabase as Backend-as-a-Service

**Decision:** Use Supabase for auth, database, storage, and real-time

**Context:**
- Need for rapid development
- PostgreSQL requirement for geospatial (PostGIS)
- Real-time requirements for messaging
- File storage for property media

**Rationale:**
- Single platform for multiple backend needs
- PostgreSQL with full SQL capabilities
- Built-in Row-Level Security (RLS)
- Real-time subscriptions out of the box
- Generous free tier for development

**Trade-offs:**
- Vendor lock-in considerations
- Scaling limits at higher tiers
- Learning curve for RLS policies

**Consequences:**
- All data access through Supabase client
- RLS policies for every table
- Real-time subscriptions for messaging
- Storage buckets for media files

### 2.3 ADR-003: TypeScript Strict Mode

**Decision:** Enable strict TypeScript configuration

**Context:**
- Large codebase (300+ files)
- Multiple developers contributing
- Need for maintainability
- Catch errors at compile time

**Rationale:**
- Type safety prevents runtime errors
- Better IDE support and autocomplete
- Self-documenting code
- Easier refactoring

**Trade-offs:**
- Slower initial development
- Migration effort for existing code
- Some third-party types incomplete

**Configuration:**
```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true
  }
}
```

### 2.4 ADR-004: Shadcn UI + TailwindCSS

**Decision:** Use Shadcn UI components with TailwindCSS

**Context:**
- Need for accessible components
- Consistent design system
- Customization requirements
- Performance considerations

**Rationale:**
- Shadcn provides accessible primitives (Radix)
- Copy-paste model allows full customization
- TailwindCSS for utility-first styling
- No runtime CSS-in-JS overhead

**Trade-offs:**
- Manual component management
- No automatic updates
- Learning TailwindCSS syntax

**Component Location:** `src/components/ui/`

### 2.5 ADR-005: AI Integration Strategy

**Decision:** Anthropic Claude as primary AI, local embeddings for search

**Context:**
- AI-powered property recommendations
- Semantic search requirements
- Cost management
- UK data residency considerations

**Rationale:**
- Claude provides high-quality reasoning
- Local embeddings (Sentence Transformers) for cost efficiency
- Hybrid approach: Claude for generation, local for search
- Circuit breaker pattern for resilience

**Trade-offs:**
- API costs for Claude
- Latency for AI operations
- Embedding storage requirements

**Implementation:**
- Claude SDK: `@anthropic-ai/sdk`
- Embeddings: `@xenova/transformers`
- Vector storage: PostgreSQL with pgvector
- Fallback: DeepSeek AI

### 2.6 ADR-006: MapTiler over Google Maps

**Decision:** Use MapTiler/MapLibre for mapping

**Context:**
- Property search with map interface
- Custom styling requirements
- Cost considerations at scale
- UK-specific map data

**Rationale:**
- Significantly lower cost than Google Maps
- Full style customization
- Open-source MapLibre renderer
- Vector tiles for performance

**Trade-offs:**
- Fewer features than Google Maps
- Less name recognition
- Some UK data quality gaps

**Implementation:**
- `@maptiler/sdk` for API
- `maplibre-gl` for rendering
- PostGIS for geospatial queries

### 2.7 ADR-007: Stripe Connect for Marketplace

**Decision:** Use Stripe Connect for service provider payments

**Context:**
- Multi-party marketplace payments
- UK regulatory compliance
- Provider payouts
- Platform fees

**Rationale:**
- Built for marketplaces
- Handles regulatory compliance
- Escrow capabilities
- Instant payouts option

**Trade-offs:**
- Commission fees (2.5% + Stripe fees)
- Provider onboarding friction
- Stripe account requirement

**Implementation:**
- Standard Connect accounts
- Platform collects 2.5% fee
- Direct charges model
- Webhook handling for events

## 7. Known Issues & Workarounds

### 7.1 Active Known Issues

#### Next.js 15 Async Params
**Issue:** Next.js 15 requires async handling of route params
**Status:** Partially migrated
**Workaround:** Use `use()` hook or async components

```typescript
// Before (Next.js 14)
export default function Page({ params }: { params: { id: string } }) {
  const { id } = params;
}

// After (Next.js 15)
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
}
```

#### Port 3000 Conflicts
**Issue:** Development server fails if port 3000 is in use
**Workaround:**
```bash
# Kill process on port 3000
npx kill-port 3000
# Or use different port
npm run dev -- -p 3001
```

#### Storybook v9 Warnings
**Issue:** Compatibility warnings with some addons
**Status:** Non-blocking, cosmetic
**Workaround:** Ignore warnings in development

### 8.2 Key API Endpoints

```
Authentication
POST   /api/auth/login
POST   /api/auth/signup
POST   /api/auth/logout
POST   /api/auth/reset-password

Properties
GET    /api/properties                   # List/search
GET    /api/properties/:id               # Get details
POST   /api/properties                   # Create (agents)
PUT    /api/properties/:id               # Update
DELETE /api/properties/:id               # Delete

Search
GET    /api/search                       # Property search
POST   /api/ai/search/semantic           # AI search
GET    /api/saved-searches               # User's saved
POST   /api/saved-searches               # Save search

Marketplace
GET    /api/v1/marketplace/providers     # List providers
POST   /api/v1/marketplace/rfq           # Create RFQ
POST   /api/v1/marketplace/quotes        # Submit quote
POST   /api/v1/marketplace/bookings      # Create booking

Messages
GET    /api/messages/conversations       # User's conversations
POST   /api/messages                     # Send message
GET    /api/messages/:conversationId     # Get messages

Admin
GET    /api/admin/users                  # List users
PUT    /api/admin/users/:id              # Update user
GET    /api/admin/verification-queue     # Verification queue
POST   /api/admin/verify/:providerId     # Approve provider
```

### 8.3 Database Key Tables

```sql
-- Users & Profiles
users                    -- Supabase auth users
user_profiles            -- Extended profile data
service_provider_profiles-- Provider-specific data

-- Properties
properties               -- Main listings table
property_media           -- Photos, videos
property_embeddings      -- AI search vectors
saved_properties         -- User favorites
viewing_schedules        -- Viewing appointments

-- Marketplace
rfq                      -- Request for quotes
quotes                   -- Provider quotes
bookings                 -- Service bookings
marketplace_reviews      -- Reviews

-- Communication
conversations            -- Message threads
messages                 -- Individual messages
notifications            -- User notifications

-- Transactions
offers                   -- Property offers
transactions             -- Sale/rental transactions
documents                -- Transaction documents

-- Admin
audit_logs               -- Activity logging
admin_actions            -- Admin operations
```
