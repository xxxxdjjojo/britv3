## 5. Auth provider (Supabase)

- **Supabase** Auth via `@supabase/ssr ^0.9.0` + `@supabase/supabase-js ^2.98.0`.
- Two client factories: `src/lib/supabase/server.ts` (Server Components / Route Handlers, cookies) and `src/lib/supabase/client.ts` (browser).
- Auth enforcement is **defence-in-depth at the layout/route level** — there is **no `middleware.ts`** (only `src/__tests__/security/middleware.test.ts`). Example guards:
  - `(protected)/layout.tsx:14-19` — `getUser()` → `redirect("/login")` if no user.
  - `(protected)/dashboard/[role]/layout.tsx` — re-checks auth + enforces URL role === `profiles.active_role`.
- Features in use: email/password, OAuth (`OAuthButtons`), email-code login (`/api/auth/email-code/*`), MFA (`/api/settings/mfa/*`, `(auth)/two-factor*`), email verification banner, account lock/suspend pages, session expiry, rate-limit page.
- `profiles.active_role` is the authoritative role source (not the URL).

| Professional | `agent` | `/dashboard/agent` |
| Professional | `service_provider` | `/dashboard/provider` |
| Professional | `mortgage_broker` | `/dashboard/broker` |

**Asymmetry (risk):** the dynamic `/dashboard/[role]/layout.tsx` `VALID_ROLES` list contains only 6 roles — it **omits `mortgage_broker`**. The broker dashboard therefore lives in a separate parallel tree `/dashboard/broker/*` rather than under `/dashboard/[role]`. Two role-routing patterns coexist (see PRODUCT_DECISION_REGISTER).

`profiles.active_role` is set during `/register/role-select` then onboarding at `/register/onboarding/[role]`. Role switching via `RoleSwitcher` component.

- **Payments:** Stripe + Stripe Connect (`@stripe/react-stripe-js`, `stripe ^20.4`); webhook at `/api/webhooks/stripe`; GoCardless mandate webhook at `/api/webhooks/gocardless`.
- **AI:** Anthropic Claude SDK (`@anthropic-ai/sdk`) + pgvector embeddings; used for property-description generation and quote drafting (`/api/ai/*`).
- **PWA:** Serwist (`@serwist/next`, `serwist`) for service worker; `web-push` for push notifications.
- **Identity verification:** configurable — env notes mention `stripe` (Stripe Identity) or `didit` as live options; "stub" mode for dev.

## 18. Notable gaps / risks (read-only observations)

1. **No `middleware.ts`** — auth is enforced layout-by-layout. Centralized route protection (e.g. matching `(protected)/*` and admin paths) is absent; relies on every page/layout repeating `getUser()`.
2. **Version drift in `CLAUDE.md`** — claims Next 16.2.1; actual is 16.2.9. Minor but undermines doc trust.
3. **Two role-routing patterns** — `/dashboard/[role]/*` for 6 roles vs. dedicated `/dashboard/broker/*` for `mortgage_broker` (omitted from `VALID_ROLES`).
4. **`britv3.0/` stub** — only 3 stray files (`britv3.0/src/types/agent.ts`, `seller.ts`, one gstack baseline). Confirmed non-functional; `tsconfig.json` excludes it.
5. **Rebrand incomplete** — 1,322 "Britestate" references remain (mostly docs/legal pages), gated by an allowlist.
6. **Schema size** — 120 migrations / ~130 tables / multiple hand-rolled `pg_cron` jobs → elevated drift/operational risk.
