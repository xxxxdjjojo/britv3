1. Description
This Epic focuses on equipping users with essential financial tools, including personalized affordability insights displayed directly on property listings. It also streamlines the offer and transaction management process and integrates secure payment and e-signature capabilities. This includes various calculators, a digital offer management system, transaction progress tracking, and payment processing.

2. Goals
To provide users with a suite of financial calculators to help them understand costs, affordability, and investment potential.
To display personalized estimated monthly mortgage payments on property listings based on user's calculator inputs, enhancing immediate affordability assessment.
To digitize and streamline the property offer and negotiation process.
To offer users a clear and transparent way to track the progress of their property transactions.
To integrate secure payment solutions for platform services, service provider fees, and potentially rent collection.
To implement e-signature functionality for key documents to expedite agreements.
3. Scope
In Scope:

Financial Calculators (all UK-focused where applicable):
Affordability Calculator: Based on income, debts, and down payment.
Mortgage Payment Calculator: Estimating monthly payments (PITI). Inputs (deposit, interest rate, term) from this calculator can be optionally saved by the user to power personalized affordability displays on listings.
Stamp Duty Land Tax (SDLT) Calculator: For England & NI, with considerations/stubs for LBTT (Scotland) and LTT (Wales).
Rent vs. Buy Calculator: Comparing costs over time.
Renovation Cost Calculator: Approximate costs for various renovation projects.
Moving Cost Calculator: Estimating costs for local/regional moves.
Personalized Affordability Display on Listings:
Option for users to save their mortgage parameters (deposit amount, interest rate, loan term) from the mortgage calculator.
Display of an estimated monthly payment next to the asking price on property search results and property detail pages, calculated using the property's price and the user's saved mortgage parameters.
Ability for users to clear or update their saved mortgage parameters.
Offer Management System:
Interface for buyers to digitally submit property offers with terms.
Functionality for sellers/agents to review, accept, reject, or counter offers.
Secure audit trail of offer history and communications related to offers.
Transaction Progress Tracking:
Visual timeline for key stages of property sale/purchase.
Checklists and informational guidance for users at each stage.
Shared view of progress for relevant parties.
Secure Payment Integration:
Integration with a payment gateway (e.g., Stripe).
Payments for service providers, platform fees, basic online rent collection.
Secure handling of payment information.
Basic invoices/receipts.
E-Signature Integration:
Integration with an e-signature service.
Preparation and sending of documents for e-signature.
Tracking of signature status and storage of signed documents.
Out of Scope: (Same as previous version of Epic 8)

Direct mortgage applications or brokerage services.
Advanced investment analysis tools beyond the provided calculators.
Full escrow account management.
Complex tax advisory services.
Direct integration with users' personal bank accounts.

7. Technical Considerations
(Additions to previous version of Epic 8)
Storing User Financial Context: Decide whether to store mortgage parameters (deposit, rate, term) in user session (temporary) or user profile (persistent for logged-in users). Ensure secure storage if persistent.
Client-Side vs. Server-Side Calculation for Listing Display:
Client-Side: When property data is fetched, the client can use the stored user parameters to calculate and display the personalized monthly payment. This is often more responsive.
Server-Side: Could be done if listings are server-rendered and user context is available. Might be less flexible if parameters change frequently.
A hybrid approach could also be considered.
Updating Property Listing Components: Modify React components for property cards and detail pages (from Epic 2) to include the new display logic for personalized payments.
Clarity of Estimates: Ensure it's very clear to the user that the displayed monthly payment on listings is an estimate based on their inputs and the property's price, and not an official quote.
Default Values: Consider what happens if the user hasn't used the mortgage calculator or saved parameters. No personalized estimate should be shown, or a generic one with clear disclaimers.
8. Design & UX Notes
(Additions to previous version of Epic 8)
Personalized Payment Display: Design how the estimated monthly payment appears on property listings (e.g., "Est. £X,XXX/month for you"). It should be prominent but clearly distinct from the asking price.
Managing Saved Inputs: Provide an intuitive way for users to see, update, or clear their saved mortgage parameters (e.g., in their profile settings or linked from the mortgage calculator).
Tooltips/Info Icons: Use tooltips to explain how the personalized estimate is calculated and that it's not a formal mortgage offer.
Opt-In: Saving financial parameters for this feature should likely be an explicit opt-in by the user for privacy reasons.
9. QA & Testing Strategy
(Additions to previous version of Epic 8)
Test Personalized Payment Display:
Verify that calculations on listings are correct based on saved user inputs and varying property prices.
Test how the display handles missing user inputs (e.g., if only deposit is saved).
Test updating and clearing saved financial parameters and ensure listing displays reflect changes.
Test across various property prices and user input scenarios.
Ensure all acceptance criteria for user stories, including the new ones, are met.

Status: To Do
Notes: New story. Logic for client-side or server-side calculation.
Acceptance Criteria:
Database schema (e.g., in profiles or a new user_financial_preferences table) allows storing user's saved deposit, interest rate, and loan term.
API endpoints (or Supabase functions) are available to save and retrieve these parameters for an authenticated user.
Frontend logic is implemented in property listing components (search results, detail page) to:

The display logic gracefully handles cases where parameters are missing or the deposit is greater than the property price.
